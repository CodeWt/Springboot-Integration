```aidl
对象的继承关系是在编译时就定义好的，所以无法在运行时改变从父类继承下来的实现。子类的实现与他的父类有着非常紧密的关系，
以至于父类实现中的任何变化必然会导致子类发生变化。当需要复用子类时，若继承下来的实现不适合解决新的问题，则父类必须重写
或被其它的适合的类替换。这种依赖关系限制了灵活性并限制了复用性。

优先使用 合成/聚合 ，而不是类的继承。
```
合成/聚合 复用原则（CARP）
```aidl
合成/聚合 复用原则 ，尽量使用 合成/聚合 ，尽量不使用类继承。

Composite 和 Aggregate 都是关联的特殊种类。合成是一种 ‘强的’ 拥有关系，体现了严格的 ‘整体与部分’ 的关系，部分和整体的生命周期一致。
聚合 是一种 ‘弱的’ 拥有关系，体现了 A对象可包含 B 对象，但B对象不是A对象的一部分。

优先使用 对象的合成聚合将有助于 保持每个类的封装性，并被集中在单个任务上，这样类和类的继承层次会保持较小的规模，并且不太可能发展为不可控的庞然大物。

```
桥接模式
```aidl
桥接模式 ， 将抽象地部分与它的实现部分分离，使它们可以独立的变化。

由于从不同的角度出发，实现的方式有多种，桥接模式的核心意图就是把这些实现独立出来，让他们各自的变化。使得每种实现的变化并不会影响其他实现，从而达到应对变化的目的。

```

何时考虑使用桥接模式
```aidl
实现一个系统或需求可能有多角度分类，每一种分类的实现都有可能变化，那么就把这种多角度实现分离出来让他们独立的变化，减少他们之间的耦合。

在发现需要多角度去分类实现对象，而只用继承会造成大量的类增加，不能满足 ‘开放-封闭’ 原则时，就应该考虑使用桥接模式了。

```
